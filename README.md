#  스마트컨트랙트 업그레이드 방식과 은행 시스템 변경 방식 비교 분석

## 1. 개요

블록체인 기술의 핵심 중 하나는 불변성이다. 하지만 소프트웨어의 결함 수정이나 기능 개선을 위해 업데이트 기능은 필수라고 할 수 있다.
기존에는 수정이 불가능하도록 설계된 스마트 컨트랙트는 기존에는 "한번 배포된 코드는 절대 바꿀 수 없다" 라는 기조로 신뢰를 형성하고, 코드의 치명적인 버그가 발견되어도 수정할 수 없어 해결책으로 사용자들이 돈을 다 빼서 새로운 주소의 컨트랙트로 이사를 가곤 했다. 이러한 스마트 컨트랙트의 한계를 극복하기 위해 스마트 컨트랙트는 우회적으로 기존의 코드가 업그레이드가 가능하도록 운영이 되고 있다.




## 2. 스마트컨트랙트의 업그레이드 방식


### 2-1. 히스토리

1. 초창기 - Code is Law 
    * 이더리움 초창기에는 한번 배포된 코드는 절대 바꿀 수 없다는 점이 블록체인의 최대 장점이자 신뢰의 근원이었다. 이는 누구도 임의로 규칙을 바꿀 수 없어야 진정한 탈중앙화라는 믿음이 강했다.
    * 코드의 치명적인 버그가 발견되면 사용자들이 돈을 다 빼서 새로운 주소의 컨트랙트로 이사를 가야 했다.

2. DAO 해킹 (2016)
    * 당시 가장 큰 프로젝트였던 'The DAO' 가 코드 취약점으로 해킹을 당해 당시가치로 750억원 가량의 이더리움이 탈취당했다.
    * 코드를 수정할 수 없었기에 하드포크를 단행했고, 이 과정에서 이더리움과 이더리움 클래식으로 블록체인이 나뉘었다.
    * 이 계기로 블록체인 전체를 뒤엎지 않고도 특정 컨트랙트의 버그만 수정할 수 있는 기술적 장치의 필요성이 대두되었다.

3. 데이터와 로직의 분리 (2017)
    * 개발자들은 코드를 직접 수정하지 않고도 업그레이드 효과를 내는 꼼수를 찾기 시작함
    * 데이터만 저장하는 컨트랙트와 로직을 담당하는 컨트랙트를 따로 분리하여 배포했다.
    * 로직 컨트랙트에 버그가 있으면 새 로직을 배포하고, 데이터 컨트랙트가 새 로직 주소를 바로보도록 설정

4. 기술적 완성 - delegatecall 과 프록시 패턴 (2018 ~ 2019)
    * 사용자는 '프록시' 컨트랙트를 호출하지만, 실제 실행은 로직 컨트랙트에서 일하고, 데이터자체는 프록시에 남는다.
    * 스마트 컨트랙트 보안 표준을 만드는 openZepplin 팀이 이 패턴을 안전하게 구현할 수 있는 라이브러리를 보급하며 대중화 됨.

5. 거버넌스와 다이아몬드 패턴 (2020 ~ )
    * 단순한 버그 수정을 넘어 거대해진 프로젝트를 효율적으로 업그레이드 하기 위한 기술 등장
    * 다이아몬드 패턴 : 컨트랙트의 용량 제한 극복을 위해 하나의 컨트랙트를 수십개의 조각으로 나눠 부분별로 업그레이드 가능하게끔 함.
    * 거버넌스 결합 : 개발자 혼자 업그레이드 하는 것이 아니라, 코인 보유자들이 투표(DAO)를 통해 업그레이드 여부를 결정하는 시스템이 결합.


### 2-2. 업그레이드 방법
프록시 패턴을 기준으로 실제로 업그레이드에 쓰이는 과정
1. 업그레이드용 코드 작성 : 로직 컨트랙트와 로직 컨트랙트를 붙일 프록시 컨트랙트를 개발.
2. 초기 배포 : 계산 로직이 담긴 로직 컨트랙트 (v1) 을 배포하고 사용자가 접속할 프록시 컨트랙트를 배포하고 v1 의 주소를 연결.
3. 새로운 버전 개발 및 테스트 : 수정할 로직이 담긴 컨트랙트를 개발. 이때 기존 데이터 저장순서를 건드리면 안됨.
4. 새 로직 배포 및 주소교체 : 새로 만든 로직 v2 를 블로겣인에 배포하고, 프록시 컨트랙트의 관리자 함수를 호출하여 v2 주소를 바로보도록 업데이트 진행.
5. 검증 및 확정 : 사용자들이 프록시 주소를 통해 거래했을 때 v2 로직이 정상 작동하는지 확인.


## 3. 기존 은행시스템과의 비교

### 3-1. 수정방식 
- 기존 은행은 서버의 소스 코드를 교체하고 재시작하지만, 스마트 컨트랙트는 새로운 컨트랙트를 배포하고 연결 주소를 변경한다.
- 기존 은행은 업데이트전 코드는 서버에서 사라지지만, 스마트컨트랙트는 프록시 컨트랙트가 호출할 로직이 변경되는거지, 과거의 로직도 블록체인에 있음.
- 은행은 은행 내부 직원이 소스코드를 수정하지만, 스마트컨트랙트는 주소 변경 권한이 누구에게 있는지 온체인에 공개된다.

### 3-2. 배포방식
- 기존 은행은 다양한 배포방식(블루그린 배포 등) 으로 트래픽을 한번에 전환하지만, 컨트랙트는 타임락 등을 통해 로직이 수정되는 시점을 설정하여 사용자에게 알린다.
- 이는 사용자들이 로직이 악의적이라면 해당 시간동안에 자기 자산을 뺄 수 있도록 보장하는 시간이라고 볼 수 있다.

### 3-3. 데이터 마이그레이션
- 기존 은행은 DB 변경 등 대규모 업데이트 시, 야간이나 주말 서비스 점검시간을 통해 데이터 일관성을 확보
- 컨트랙트는 아래 방식을 통해 마이그레이션 진행
    - 일괄 전송 방식 : 관리자가 구 버전의 데이터를 모두 읽어서 신버전 컨트랙트에 옮기는 방식
    - 사용자 요청 시 전송 방식 : 사용자가 신 버전 컨트랙트를 처음 사용할때 구버전에서 자신의 데이터를 스스로 가져오게끔 만드는 방식
    - 스냅샷 방식 : 특정 시점에 구 버전 데이터를 추출한 뒤 데이터를 바탕으로 신 버전 컨트랙트 배포.

### 3-4. 핵심 비교 분석

  ┌────────────────┬────────────────────────────────────────────────┬─────────────────────────────────────────┐
  │ 비교 항목      │ 스마트 컨트랙트 업그레이드                     │ 은행 시스템 변경                        │
  ├────────────────┼────────────────────────────────────────────────┼─────────────────────────────────────────┤
  │ 기본 철학      │ 불변성 유지 및 우회적 업데이트                 │ 안정적 서비스 유지 및 철저한 검증       │
  │ 데이터 관리    │ 상태(State)와 로직의 분리 필수                 │ 관계형 DB(RDBMS) 스키마 관리 중심       │
  │ 변경 권한      │ 거버넌스 투표 또는 스마트 컨트랙트 소유자      │ 내부 결재 라인 및 IT 운영팀             │
  │ 롤백(Rollback) │ 매우 어려움 (이전 프록시 설정으로 되돌려야 함) │ 용이함 (DB 백업 및 스냅샷 활용)         │
  │ 투명성         │ 누구나 코드 변경 및 시점 확인 가능             │ 폐쇄적 (내부 시스템 로그로만 확인 가능) │
  │ 비용           │ 온체인 배포 및 트랜잭션 비용(Gas) 발생         │ 인프라 유지보수 및 인력 비용 발생       │
  └────────────────┴────────────────────────────────────────────────┴─────────────────────────────────────────┘


## 4. 결론

스마트 컨트랙트는 신뢰의 주체를 코드로 옮기려는 시도로 볼 수 있다. 따라서 업그레이드 과정 조차 코드와 거버넌스에 의해 통제된다.

그에 반면 기존 은행 시스템은 기관의 신뢰를 바탕으로 하며, 사고 방지를 위한 인적,절차적 통제에 집중한다.

은행은 이 두 방식의 장점을 결합하여 아래 두가지 목표를 이룰 수 있을지 확인해봐야 할 것 같다.
* 블록체인의 투명한 이력관리 도입하여 감사 추적성을 높일 수 있다.
* 블록체인 도입 시 은행의 엄격한 테스트 및 보안 심사 과정을 도입하여 프록시 취약점으로 인한 해킹 사고를 방지한다.
